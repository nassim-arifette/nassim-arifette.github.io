In the usual categorical semantics of the simply typed lambda calculus, types
are objects and terms are morphisms. Products interpret product types, while
exponentials interpret function types. This is why **cartesian closed
categories** are a natural semantic home for typed functional programs.

There is, however, a deliberate loss of information. The equations of the
lambda calculus—including $\beta$ and $\eta$—are interpreted as equalities of
morphisms. Consider the familiar reduction

$$ (\lambda x.M)\,N \longrightarrow_\beta M[x := N]. $$

In a one-dimensional categorical model, the term before evaluation and the term
after evaluation denote the same arrow. The model tells us what the program
means, but not **how it computed that meaning**. The redex, the rewrite that
contracts it, and every intermediate program state have disappeared.

Robert A. G. Seely's proposal in *Modelling Computations: A 2-Categorical
Framework* (LICS 1987) is to keep this computational dimension. Reductions and
conversions are promoted from equations to structure.

## From equality to a second dimension

A strict 2-category has three levels:

- **objects**, which we read as types;
- **1-morphisms**, which we read as typed terms in context;
- **2-morphisms**, which we read as reductions or conversions between terms.

For two programs $f,g : A \to B$, a computation

$$ \alpha : f \Rightarrow g $$

is not merely the assertion $f=g$. It is a directed witness that $f$ computes
to $g$. Two programs can have the same input and output behaviour while
retaining different computational histories.

There are two ways to compose these computations:

1. **Vertical composition** chains reductions. If
   $\alpha:f\Rightarrow g$ and $\beta:g\Rightarrow h$, then
   $\beta\circ\alpha:f\Rightarrow h$.
2. **Horizontal composition** places a reduction inside a larger program
   context. If a subterm reduces, the term containing it can reduce as well.

The interchange law states that these two forms of composition cooperate. It
is the algebraic version of an important implementation principle: local
rewrites remain valid inside larger programs.

## Why Haskell for the executable model?

Lean or Agda would be the strongest choice for machine-checking every theorem.
For an explanatory implementation, Haskell offers a better balance: algebraic
data types match the syntax of the lambda calculus, values are immutable,
pattern matching follows the inference rules, and the host compiler checks the
model itself.

The code below is still a small **syntactic illustration**, not Seely's full
categorical construction. Unlike the earlier Python version, however, the
object language is explicitly typed.

### Types, terms, and contexts

We start with base types and function types. A lambda binder carries its input
type, which lets us check a term-in-context $\Gamma\vdash M:A$.

```haskell
{-# LANGUAGE LambdaCase #-}

module Main where

import Data.List (intercalate)
import qualified Data.Map.Strict as Map
import qualified Data.Set as Set

data Type
  = Base String
  | Type :-> Type
  deriving (Eq)

infixr 5 :->

instance Show Type where
  show (Base name) = name
  show (domain :-> codomain) = "(" ++ show domain ++ " → " ++ show codomain ++ ")"

data Term
  = Var String
  | Lam String Type Term
  | App Term Term
  deriving (Eq)

pretty :: Term -> String
pretty = \case
  Var name -> name
  Lam parameter _ body -> "(λ" ++ parameter ++ "." ++ pretty body ++ ")"
  App function argument -> "(" ++ pretty function ++ " " ++ pretty argument ++ ")"

instance Show Term where
  show = pretty

type Context = Map.Map String Type

infer :: Context -> Term -> Either String Type
infer context = \case
  Var name ->
    maybe (Left ("unbound variable: " ++ name)) Right (Map.lookup name context)

  Lam parameter parameterType body -> do
    bodyType <- infer (Map.insert parameter parameterType context) body
    pure (parameterType :-> bodyType)

  App function argument -> do
    functionType <- infer context function
    argumentType <- infer context argument
    case functionType of
      domain :-> codomain
        | domain == argumentType -> Right codomain
        | otherwise -> Left "function and argument types do not match"
      _ -> Left "attempted to apply a non-function"
```

The constructor `(:->)` is not decoration: ill-typed applications are rejected
before evaluation. This keeps the first categorical dimension—the source and
target types—visible in the program.

### Capture-avoiding substitution

A beta step replaces the bound variable with the argument. Naive textual
replacement is wrong because it may **capture** a free variable. Substituting
$y$ for $x$ in $\lambda y.x$, for example, must first rename the bound $y$.

```haskell
freeVariables :: Term -> Set.Set String
freeVariables = \case
  Var name -> Set.singleton name
  Lam parameter _ body -> Set.delete parameter (freeVariables body)
  App function argument -> freeVariables function `Set.union` freeVariables argument

allNames :: Term -> Set.Set String
allNames = \case
  Var name -> Set.singleton name
  Lam parameter _ body -> Set.insert parameter (allNames body)
  App function argument -> allNames function `Set.union` allNames argument

freshName :: String -> Set.Set String -> String
freshName base forbidden = choose (0 :: Int)
  where
    choose index =
      let candidate = if index == 0 then base else base ++ "_" ++ show index
       in if candidate `Set.member` forbidden then choose (index + 1) else candidate

substitute :: Term -> String -> Term -> Term
substitute term variable replacement = case term of
  Var name
    | name == variable -> replacement
    | otherwise -> term

  App function argument ->
    App
      (substitute function variable replacement)
      (substitute argument variable replacement)

  Lam parameter parameterType body
    | parameter == variable -> term
    | parameter `Set.notMember` freeVariables replacement ->
        Lam parameter parameterType (substitute body variable replacement)
    | otherwise ->
        let forbidden =
              allNames body
                `Set.union` allNames replacement
                `Set.union` Set.singleton variable
            renamed = freshName parameter forbidden
            alphaConverted = substitute body parameter (Var renamed)
         in Lam
              renamed
              parameterType
              (substitute alphaConverted variable replacement)
```

The `allNames` check is intentional. Looking only at free variables while
renaming a binder can choose a name already bound deeper in the term and cause a
second, subtler capture.

### Reductions as first-class values

An ordinary evaluator returns the next term and throws the transition away.
Here it returns a `Reduction` containing its source, target, rule, and the
position of the contracted redex. This value plays the role of a generating
2-cell.

```haskell
data Direction = InFunction | InArgument | InBody
  deriving (Eq, Show)

data Reduction = Reduction
  { source :: Term,
    target :: Term,
    rule :: String,
    position :: [Direction]
  }
  deriving (Eq)

instance Show Reduction where
  show reduction =
    pretty (source reduction)
      ++ " --"
      ++ rule reduction
      ++ "@"
      ++ location
      ++ "→ "
      ++ pretty (target reduction)
    where
      location = case position reduction of
        [] -> "root"
        path -> intercalate "/" (map show path)

betaStep :: Term -> Maybe Reduction
betaStep term = case term of
  App (Lam parameter _ body) argument ->
    Just (Reduction term (substitute body parameter argument) "β" [])

  App function argument ->
    case betaStep function of
      Just step ->
        Just
          ( Reduction
              term
              (App (target step) argument)
              (rule step)
              (InFunction : position step)
          )
      Nothing -> case betaStep argument of
        Just step ->
          Just
            ( Reduction
                term
                (App function (target step))
                (rule step)
                (InArgument : position step)
            )
        Nothing -> Nothing

  Lam parameter parameterType body ->
    case betaStep body of
      Just step ->
        Just
          ( Reduction
              term
              (Lam parameter parameterType (target step))
              (rule step)
              (InBody : position step)
          )
      Nothing -> Nothing

  Var _ -> Nothing
```

The strategy is normal order: it contracts the leftmost, outermost redex first.
Other strategies produce different paths through the same space of terms.

The recursive cases also expose horizontal structure. A reduction
$\alpha:M\Rightarrow M'$ induces, among others,

$$ \alpha\,N : M\,N \Rightarrow M'\,N $$

and

$$ \lambda x.\alpha : \lambda x.M \Rightarrow \lambda x.M'. $$

In the code, `position` records the surrounding program context in which the
local reduction occurs.

### Vertical composition as an execution trace

Successive 2-cells can now be collected instead of discarded. Composition is
partial: the target of the first trace must equal the source of the second.

```haskell
data Trace = Trace
  { start :: Term,
    steps :: [Reduction]
  }
  deriving (Eq, Show)

end :: Trace -> Term
end (Trace initial []) = initial
end (Trace _ reductions) = target (last reductions)

thenTrace :: Trace -> Trace -> Either String Trace
thenTrace first second
  | end first == start second =
      Right (Trace (start first) (steps first ++ steps second))
  | otherwise = Left "cannot compose traces with different boundaries"

normalize :: Int -> Term -> Either String Trace
normalize maximumSteps initial = go maximumSteps initial []
  where
    go remaining current history = case betaStep current of
      Nothing -> Right (Trace initial (reverse history))
      Just step
        | remaining == 0 -> Left "reduction limit reached"
        | otherwise -> go (remaining - 1) (target step) (step : history)
```

Finally, we type-check and evaluate

$$ (\lambda x.x)\big((\lambda y.y)\,z\big). $$

```haskell
main :: IO ()
main = do
  let valueType = Base "A"
      context = Map.singleton "z" valueType
      identity name = Lam name valueType (Var name)
      program = App (identity "x") (App (identity "y") (Var "z"))

  programType <- either fail pure (infer context program)
  trace <- either fail pure (normalize 1000 program)

  putStrLn ("type: " ++ show programType)
  putStrLn ("start: " ++ pretty (start trace))
  mapM_ print (zipWith (,) [(1 :: Int) ..] (steps trace))
  putStrLn ("normal form: " ++ pretty (end trace))
```

Running the complete file produces:

```text
type: A
start: ((λx.x) ((λy.y) z))
(1,((λx.x) ((λy.y) z)) --β@root→ ((λy.y) z))
(2,((λy.y) z) --β@root→ z)
normal form: z
```

A conventional denotational model identifies the first term, the intermediate
term, and `z` when the calculus declares them equal. The trace instead retains
the directed composite

$$ ((\lambda x.x)((\lambda y.y)z)) \Rightarrow ((\lambda y.y)z) \Rightarrow z. $$

The endpoints give the extensional result; the composite gives the intensional
history.

## Different paths are meaningful

Now consider a term with two independent redexes:

$$ ((\lambda x.x)\,a)\;((\lambda y.y)\,b). $$

We may reduce the function side first or the argument side first. Both paths
reach $a\,b$, but they are different sequences of steps.

```text
((λx.x) a) ((λy.y) b)  ──function──▶  a ((λy.y) b)
          │                                  │
       argument                           argument
          │                                  │
          ▼                                  ▼
((λx.x) a) b             ──function──▶      a b
```

The commuting square is the computational face of interchange: independent
local reductions can be composed in either order. In systems with effects,
resources, concurrency, or non-confluence, retaining path information becomes
even more important because evaluation order may change cost, observable
behaviour, or the result itself.

## Where the analogy stops

The Haskell program makes the main distinction executable, but it is not a full
construction of Seely's 2-category.

- It checks simply typed terms and generates beta reductions, but it does not
  implement products, eta conversion, or all structural rules.
- It chooses one evaluation strategy instead of constructing every possible
  reduction between two terms.
- It does not quotient reduction paths by the equations required for 2-cells or
  prove the interchange law.
- It does not construct 2-adjunctions or the lax universal properties in
  Seely's development.
- Named variables keep the presentation readable, but a formal development
  would usually represent binding with de Bruijn indices or a verified binding
  library.

For machine-checked metatheory, Lean or Agda is the natural next step. For seeing
why a computation should be data rather than equality, this typed Haskell model
keeps the structure visible without burying it under proof machinery.

## Why the 2D view matters

The move from categories to 2-categories changes the semantic question. Instead
of asking only

> What morphism does this program denote?

we can also ask

> Which computation transforms this program into that one, and how do such
> computations compose?

That second question matters in compiler optimisation, proof transformation,
rewriting systems, parallel evaluation, and provenance-aware pipelines. Each
involves not just objects and maps, but **maps between maps**.

Seely's insight is simple to state and far-reaching in consequence: do not
force computation to become equality too early. Keep the path.
